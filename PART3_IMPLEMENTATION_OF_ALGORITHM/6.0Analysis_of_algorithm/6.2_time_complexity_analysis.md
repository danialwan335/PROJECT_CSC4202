# 6.2 Time Complexity Analysis

This section analyses the time complexity of each component of the Dijkstra's Algorithm implementation and then derives the overall complexity.

---

## Component-by-Component Analysis

### 1. Graph Construction

```java
for each edge (u, v, w):
    adj[u].add({v, w})
    adj[v].add({u, w})
```

Processing E edges, each taking O(1) time:
> **T₁ = O(E)**

### 2. Initialisation

```java
Arrays.fill(dist, MAX_VALUE);  // O(V)
Arrays.fill(prev, -1);         // O(V)
pq.offer({0, source});         // O(log 1) = O(1)
```

> **T₂ = O(V)**

### 3. Main Dijkstra Loop

The main loop processes each vertex and each edge:

**Node extractions:** Each vertex is extracted from PQ at most once (due to visited[] check). Each extraction from a binary heap PQ takes O(log V).
> Total extraction cost: **O(V log V)**

**Edge relaxations:** Each edge (u, v) is relaxed at most twice (once from u's side, once from v's side — since the graph is undirected). Each relaxation may trigger a PQ insertion (O(log V)).
> Total relaxation cost: **O(E log V)**

**Total main loop:**
> **T₃ = O(V log V + E log V) = O((V + E) log V)**

### 4. Path Reconstruction

```java
for (int cur = dest; cur != -1; cur = prev[cur])
    path.addFirst(cur);
```

This traces back at most V nodes:
> **T₄ = O(V)**

### 5. Output / Segment Printing

Printing at most V nodes in the path and all V distance entries:
> **T₅ = O(V)**

---

## Total Time Complexity

Combining all components:

> **T = T₁ + T₂ + T₃ + T₄ + T₅**
> **T = O(E) + O(V) + O((V+E) log V) + O(V) + O(V)**
> **T = O((V + E) log V)**

Since O((V+E) log V) dominates all other terms, the **overall time complexity is O((V + E) log V)**.

---

## Case Analysis

### Best Case — Ω((V + E) log V)

Even in the most favourable arrangement (e.g., the optimal path is found immediately from the first extraction), Dijkstra's Algorithm must still:
- Extract all vertices from PQ: O(V log V)
- Visit all edges for relaxation: O(E log V)

The algorithm has no early exit mechanism that avoids processing all reachable nodes — even with only 1 optimal path, all other edges must be considered to confirm no shorter path exists.

> **Best Case: Ω((V + E) log V)** — same asymptotic bound as worst case.

In practice, for our 8-node, 9-edge scenario, this terminates in microseconds.

### Average Case — Θ((V + E) log V)

In the average case, the distribution of edge weights and graph structure does not change the asymptotic bound — every vertex is still extracted once and every edge relaxed at most twice.

> **Average Case: Θ((V + E) log V)**

### Worst Case — O((V + E) log V)

The worst case occurs when:
- The graph is dense (E ≈ V²) — every node has edges to every other node.
- Many PQ insertions occur due to multiple distance updates for the same node.

For a dense graph: O((V + V²) log V) = **O(V² log V)**.

For our sparse post-landslide network (V = 8, E = 9): O((8 + 9) log 8) = O(17 × 3) ≈ **O(51 operations)** — effectively instantaneous.

> **Worst Case: O((V + E) log V)** → O(V² log V) for dense graphs.

## Space Complexity

| Data Structure | Space |
|---|---|
| Adjacency list | O(V + E) |
| dist[] array | O(V) |
| prev[] array | O(V) |
| visited[] array | O(V) |
| Priority Queue | O(V + E) worst case (stale entries) |
| **Total Space** | **O(V + E)** |

---

## Comparison with Alternative Algorithms

| Algorithm | Time Complexity | Optimal for Non-Neg Weights? |
|---|---|---|
| Dijkstra (binary heap) | O((V+E) log V) | ✅ Yes |
| Bellman-Ford (DP) | O(VE) | ✅ Yes (but slower) |
| Floyd-Warshall | O(V³) | ✅ Yes (all-pairs, overkill) |
| BFS | O(V + E) | ❌ Unweighted only |
| Greedy (naïve) | O(E) | ❌ Not guaranteed optimal |

Dijkstra's Algorithm with a binary heap is the **most efficient correct algorithm** for this single-source, non-negative weight shortest path problem.

---

## Potential Improvement

For very large road networks (V > 10,000), a **Fibonacci heap** reduces Dijkstra's complexity to O(E + V log V), eliminating the log factor from edge relaxations. However, for our scenario (V = 8, E = 9), the binary heap implementation is already optimal in practice.

Additionally, an **A\* (A-star) search** could speed up computation by using GPS coordinates as a heuristic to guide the search toward the destination. This is how Google Maps and Waze achieve real-time routing — but the underlying shortest path guarantee remains Dijkstra-equivalent.
