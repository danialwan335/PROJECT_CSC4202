# 2.3 Constraints and Challenges

## Constraints

### 1. Non-Negative Edge Weights
All travel times must be non-negative (≥ 0). This is a fundamental constraint because Dijkstra's Algorithm is only correct for graphs with non-negative weights. In our scenario, this is naturally satisfied — a road cannot be traversed in negative time. However, it also means Bellman-Ford (which handles negative weights) is an acceptable but slower alternative.

### 2. Graph Connectivity
The algorithm assumes at least one path exists from the hospital to the village through the surviving road network. If the landslide has completely cut off all possible routes (fully disconnected graph), Dijkstra's Algorithm will correctly report that no path exists (distance = infinity), signalling the need for helicopter evacuation or other emergency measures.

### 3. Static Graph Assumption
The model assumes the road network does not change during the algorithm's execution — i.e., no additional roads collapse while the ambulance is en route. This is a necessary simplification; in practice, real-time updates would require a dynamic shortest path algorithm (not implemented here).

### 4. Discrete Node Representation
The model requires all relevant junctions and landmarks to be explicitly represented as nodes. Roads that branch off between two defined nodes are not captured unless an intermediate node is added. This means the graph must be carefully constructed from real road network data.

### 5. Single-Source, Single-Destination
The current algorithm computes the shortest path from one source (hospital) to one destination (village). For multi-casualty events requiring multiple simultaneous evacuations from different points, the algorithm must be re-run for each source node.

## Challenges

### 1. Accurate Edge Weight Estimation Under Disaster Conditions
Travel times on logging roads post-landslide are difficult to estimate accurately. Partial debris, night-time visibility, and mud make real speeds highly variable. Incorrect weights lead to suboptimal routing. In practice, this requires real-time input from scouts or drones before the algorithm is run.

### 2. Incomplete or Missing Road Data
Not all secondary and logging roads in highland Selangor are mapped in standard GIS databases. The graph may be incomplete, missing edges that actually exist. This requires ground-truthing or local knowledge to supplement digital map data.

### 3. Blocked Road Detection
The algorithm must know which roads are blocked (edges to be removed from the graph) before it runs. Identifying blocked roads in real-time requires communication with field teams, which may be disrupted during the disaster.

### 4. Route Navigability vs. Route Optimality
The shortest-time route may pass through roads that, while technically passable, are unsafe for a heavy ambulance vehicle (e.g., a bridge with insufficient load rating, or a road too narrow for a full-size ambulance). The algorithm does not inherently account for vehicle-specific constraints unless such segments are removed from the graph manually before computation.

### 5. Human Override Necessity
The algorithm provides an optimal solution based on its input data. If field conditions differ from the modelled data (e.g., a road appears clear in the model but is actually flooded), the ambulance driver must be empowered to override the algorithm's recommendation and take an alternative route. The algorithm is a decision support tool, not an infallible command.
